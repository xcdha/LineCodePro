# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

LineCode Pro (`cn.lineai`, applicationId `cn.lineai`) is an Android app that hosts an in-device AI coding workspace: an LLM chat with a tool-call loop over user-selected project trees, plus optional SSH/Termux integration. Multi-module Gradle project (see `settings.gradle.kts`), pure Java sources (no Kotlin in `app/src`), Android Gradle Plugin 9.2.0, `compileSdk = 36`, `minSdk = 24`, `targetSdk = 36`, Java 11.

## Module layout

Root project in `settings.gradle.kts` includes:
- `:build-logic` (composite build) — `linecode.convention` plugin shared by all modules
- `:core-model` (java-library) — data classes / DTOs / enums (`ModelConfig`, `ThemePalette`, `OutputSettings`, `ContextSizeParser`, `ModelContextParser`, `ChatMessage`, ...)
- `:core-api` (java-library) — interface abstractions (`ToolInfo`, `ToolNames`, `ToolCategory`, `ToolDisplayCategory`, `ModelServiceProvider`)
- `:core-security` (java-library) — `UrlPolicy`, `SimpleHttpClient`
- `:ui-theme` (Android Library) — `LineTheme` and `cn.lineai.ui.theme.*` (shared UI infrastructure referenced by ~80 files)
- `:markdown` (Android Library) — Markdown rendering (`cn.lineai.ui.markdown.*`, depends on commonmark + GFM tables); depends on `:core-model`, `:ui-theme`
- `:data` (Android Library) — repositories / DB / stores (`LineCodeDatabase`, `LineCodeSchema`, `cn.lineai.data.repository.*`)
- `:feature-tool` (Android Library) — `BaseTool`, `ToolRegistry`, built-in tools, `FileToolPathPolicy`, `ToolContext`
- `:feature-model` (Android Library) — `ModelProtocol`, `ModelProtocolFactory`, protocols (`OpenAiCompatibleProtocol`, `CodexResponsesProtocol`, `AnthropicMessagesProtocol`, `LocalGgufProtocol`), `ContextManager`, `ContextCompactionService`
- `:feature-ssh` (Android Library) — `SshService`, `SshFileTreeRepository` (jsch)
- `:feature-share` (Android Library) — export/share/PDF
- `:terminal-provider` / `:ipc` — Termux integration
- `:app` (Android Application) — `MainActivity`, `MainCoordinator`, controllers (`cn.lineai.mvp.*`), UI components (`cn.lineai.ui.*`), tools wiring

When adding code, choose the lowest-level module that fits: DTOs → `:core-model`; interfaces → `:core-api`; UI infrastructure → `:ui-theme`; a new tool → `:feature-tool`; a new protocol → `:feature-model`. Do not reach back into `:app` from a library module. `ai ↔ tool` circular dependency is broken: `tool` must not import `cn.lineai.ai`, and `ai` may only reference `ToolInfo`/`ToolNames`/`ToolCategory` from `:core-api`.

## Commands

All Gradle commands go through the wrapper. Settings force `FAIL_ON_PROJECT_REPOS`, and the repo is wired to a Tencent Cloud Maven mirror — keep that in `settings.gradle.kts` when adding repositories.

- Debug build: `./gradlew :app:assembleDebug`
- Debug build signed with debug cert for sideload (separate APK output): `./gradlew :app:assembleDebugUserCert` (also `exportDebugUserCertApk` copies it to `app/build/outputs/apk/debugUserCert/export/LineCode-user-cert-debug.apk`)
- Release build: `./gradlew :app:assembleRelease` — fails fast via `validateReleaseSigning` unless `signing.properties` (gitignored) contains `storeFile`, `storePassword`, `keyAlias`, `keyPassword`. On success, `exportReleaseApk` renames the artifact to `LineCode Pro <versionName>.APK` under `app/release/`, then `purgeReleaseSymbolFiles` deletes `outputs/mapping/release` and `outputs/native-debug-symbols/release`. The release uses an 8192-entry obfuscation dictionary generated by `generateReleaseObfuscationDictionary` into `app/build/generated/r8/obfuscation-dictionary.txt`.
- Unit tests: `./gradlew :app:testDebugUnitTest`
- Single test: `./gradlew :app:testDebugUnitTest --tests "cn.lineai.tool.ToolRegistryTest"` (append `.methodName` for one method)
- Per-module build/test: `./gradlew :feature-tool:testDebugUnitTest`, `./gradlew :ui-theme:build`, `./gradlew :markdown:build`, etc.
- Lint: `./gradlew :app:lintDebug` (rules in `app/lint.xml`)

Bumping the version requires editing `releaseVersionName` and `defaultConfig.versionCode` in `app/build.gradle.kts`; the release APK filename is derived from `releaseVersionName`.

## Architecture

### MVP coordinator
`MainActivity` is the only Activity. It instantiates `MainCoordinator` (the presenter), constructs `MainChatView` programmatically (UI is built in Java code, not inflated from XML — `lint.xml` silences `ViewConstructor` accordingly), and forwards SAF picker / permission results. The coordinator implements `MainUiController` and delegates large chunks of behavior to per-concern controllers in `cn.lineai.mvp.*` (e.g. `ChatController`, `ChatInteractionController`, `GenerationFlowController`, `ToolRunController`, `ModelManagementController`, `SettingsManagementController`, `ScreenNavigationController`, `PermissionModeController`, `LineCodeArchiveController`, `ProjectSheetController`, `SshFileTreeController`, `FileOperationController`, `AgentExecutionController`, `ContextCompactionController`, `PhoneControlController`, `ErrorLogController`, `StorageMaintenanceController`). `MainDependencies` / `MainControllerInitializer` wire repositories, services, and the tool stack into the coordinator. UI state flows through `ChatUiStateAssembler` → `ChatUiState` → `MainContract.View.render(...)`. When adding new chat/tool features, prefer extending an existing controller and threading state through the assembler rather than reaching into views directly.

### Model protocol abstraction
`cn.lineai.ai.protocol.ModelProtocol` is the streaming/completion interface. `ModelProtocolFactory` dispatches on `ModelProtocolType`:
- `OPENAI_COMPATIBLE` → `OpenAiCompatibleProtocol` (default; also see `OpenAiCompatibleCapabilities`, `OpenAiResponsesCompactionProtocol`)
- `CODEX_RESPONSES` → `CodexResponsesProtocol` (uses `ResponsesInputBuilder`)
- `ANTHROPIC_MESSAGES` → `AnthropicMessagesProtocol`
- `LOCAL_GGUF` → `LocalGgufProtocol`

Messages flow via `cn.lineai.ai.message.*` (`SystemModelMessage`, `UserModelMessage`, `AssistantModelMessage`, `ToolModelMessage`). `ModelClient` is the high-level entry point used by `MainCoordinator`. Tool-call text inside a stream is parsed by `ToolCallTextParser`; reasoning by `ThinkTagParser`. System prompts are assembled by `cn.lineai.ai.prompt.SystemPromptProvider` from `app/src/main/assets/prompts/*.txt` templates (`system-prompt-template.txt`, tone variants, `context-compaction-template.txt`, `memory-extraction-template.txt`, `skill-extraction-template.txt`, `work-directory-template.txt`, `learning-context-template.txt`, `model-identity-template.txt`) — modify templates there, not by hardcoding in Java. `SystemPromptProvider.build(..., ModelConfig model)` renders the model identity block (modelId / name / provider / protocol) from the currently selected model and injects it as `{{MODEL_IDENTITY}}` so the assistant can answer capability questions against the actual model id.

### Model config & context size
`ModelConfig` (`:core-model`) carries `contextSize` (tokens) as a first-class field, parsed by `ContextSizeParser` from user input like `128K`, `1m`, `128000`. UI uses a plain text field (`R.string.model_field_context_size`); the legacy `{id}[{size}]` suffix in `modelId` is still tolerated for old DB rows via `ModelContextParser.parse(ModelConfig)` / `apiModelId(ModelConfig)`. Protocols must call `apiModelId(ModelConfig)` rather than reading `modelId` directly, so the `[size]` suffix is stripped when needed. The `model_configs` SQLite table has a `context_size` column; `ModelRepository.ensureModelConfigColumns` auto-`ALTER`s old databases.

### Tool system
`BaseTool` (name, description, category, JSON schema, `execute(JSONObject, ToolContext)`) is the contract; subclasses also expose `needsConfirmation()`, `isAllowedInReadonlyMode()`, `promptSupplement()`, and a `public static final NAME` constant. `ToolRegistry` registers built-ins in `cn.lineai.tool.builtin.*` at construction (`FileReadTool`, `FileWriteTool`, `FileEditTool`, `FileDeleteTool`, `GlobTool`, `ListDirectoryTool`, `AgentTool`, `AgentPipelineTool`, `ShellExecuteTool`, `ImageUnderstandingTool`, `ImageGenerationTool`, `WebSearchTool`, `WebFetchTool`) and then loads user extensions with `agentx_` / `mcpx_` name prefixes via `reloadExtensions()` (sourced from `ExtensionRepository` → `CustomAgentExtensionTool`, `CustomMcpHttpTool`). Tool execution is driven by `ToolExecutionCoordinator` + `ToolExecutor`; user confirmation gating goes through `PermissionModeController` and `ToolReviewListener`, with session-scoped auto-confirmation tracked in `MainCoordinator.sessionAutoConfirmedTools`. File-touching tools route paths through `FileToolPathPolicy`, which consults `ToolContext.isBypassPathProtection()` (see Security below) before enforcing the workspace boundary.

### Context window management (dynamic compaction)
`ContextManager` estimates token usage with the constants `CHARS_PER_TOKEN = 4` and `MESSAGE_OVERHEAD_TOKENS = 8`, reserving `DEFAULT_RESERVE_TOKENS = 2048`. `ContextCompactionService` (`:feature-model`) implements **dynamic compaction** inspired by codex / Claude-Code rather than a single hard threshold:
- Soft trigger at 50% of context window (`SOFT_COMPACT_TRIGGER_RATIO`): the earliest 70% of compactable messages are summarised while the most recent 30% (`SOFT_COMPACT_TAIL_KEEP_RATIO`) are preserved verbatim. Result layout: `[head(excludeFromContext)] + [summary] + [tail(original)] + [preservedTail] + [progressDone]`.
- Hard trigger at 80% (`shouldCompact`): full compaction fallback.
- Transcript is built segment-by-segment (`List<String>`, each ≤ `TRANSCRIPT_SEGMENT_MAX_CHARS = 256KB`) to avoid single-`StringBuilder` OOM. **Do not reintroduce per-message or total-length truncation** — earlier `MAX_MESSAGE_CONTENT_CHARS` / `MAX_TRANSCRIPT_CHARS` limits were removed as harmful. `ContextCompactionController.startSoftContextCompaction` / `finishSoftContextCompaction` drive the flow; `ChatInteractionController` checks the soft trigger before each request when the hard trigger has not fired. `OutOfMemoryError` is caught in both the service and the controller as a last-resort fallback.
- `AndroidManifest.xml` sets `android:largeHeap="true"` to support large transcripts.
- `MemoryExtractionService` and `LearningContextRepository` feed durable knowledge back into the system prompt.

### Image attachments in chat
`ComposerView` exposes a dedicated image button (right of the attach `+`) that opens the system image picker (`ACTION_OPEN_DOCUMENT`, `image/*`). Selected images are compressed (long edge ≤ 1568px, JPEG q=85, >3.5MB further downgraded) and base64-encoded. Send path goes through `ChatInteractionController.sendMessageWithImage` → `ImageInputPayload.rawInputJson(prompt, mimeType, base64)` → `ChatMessage.responseInputItemJson`. Protocols detect `ImageInputPayload.KIND` in `message.getRawInputJson()` and emit the correct shape per provider (OpenAI `image_url`, Anthropic `image.source.base64`, Codex `input_image`). No protocol changes are needed when adding new image entry points — reuse `ImageInputPayload`.

### Storage
SQLite is centralised in `cn.lineai.data.db.LineCodeDatabase` (singleton, version 1) with schema in `LineCodeSchema` (tables for projects, models, conversations, messages, blocks, tool calls/results, attachments, diffs, memories, working memory, skills/usage, extension agents/MCPs, import jobs). Repositories in `cn.lineai.data.repository.*` are the only callers — do not open the DB directly from controllers. Settings repositories (`SettingsRepository`, `InputSettingsRepository`, `OutputSettingsRepository`, `ThemeSettingsRepository`, `AiBehaviorSettingsRepository`, `ToolSettingsRepository`, etc.) follow the same pattern. `ThemeSettingsRepository` returns a `ThemePalette` and never imports `cn.lineai.ui.theme.LineTheme`; the caller (`MainDependencies` / `MainActivity` / `RepositorySettingsStore`) invokes `LineTheme.apply(palette)` so the data layer stays UI-free.

### Project workspace & files
Workspace roots come from user-picked SAF trees. `WorkspacePaths` resolves on-device paths, `SafPathResolver` translates between SAF document URIs and POSIX-like paths, and `StoragePermissionManager` drives the `MANAGE_EXTERNAL_STORAGE` / legacy storage prompts (the manifest requests both because the app operates as a coding workspace over external project trees). SSH access goes through `SshService` (jsch) and `SshFileTreeRepository`.

### Security / URL policy
All outbound URLs (in-app browser, external link opening, model HTTP, model catalog, web fetch/search, custom MCP HTTP) must pass through `cn.lineai.security.UrlPolicy`. HTTP-only protocols, and cleartext is restricted to `localhost`, `127.0.0.1`, `10.0.2.2` (mirrored in `res/xml/network_security_config.xml`, `cleartextTrafficPermitted="false"` by default). The built-in WebView disables file/content access, blocks mixed content, and keeps JavaScript off unless the user explicitly toggles it. When exporting `.linecode` archives, `ArchiveSecretRedactor` strips model API keys, SSH passwords / private keys / passphrases, web search API keys, and sensitive MCP headers — extend it whenever you add a new secret-bearing field. See `docs/android-compliance-audit.md` for the documented threat model and accepted risks.

The Security screen exposes **Bypass path protection** (`OutputSettings.bypassPathProtection`, default off, key `@lineai_bypass_path_protection`). When enabled, `FileToolPathPolicy.resolve(ToolContext, ...)` skips the workspace-boundary check and returns the canonical path directly. `ToolContext.bypassPathProtection` is propagated by `ToolExecutor.injectDependencies` and `AgentExecutionController.executeAgentToolCall`, and `AgentExecutionController.validateAgentWriteScope` skips the `write_scope` check when bypass is on. Enabling the toggle shows a warning dialog; the user must confirm. All file-touching tools (`FileReadTool`/`FileWriteTool`/`FileEditTool`/`FileDeleteTool`/`GlobTool`/`ListDirectoryTool`/`ImageUnderstandingTool`) honour this flag automatically because they go through `FileToolPathPolicy`.

### Markdown & custom views
The chat list renders Markdown through `:markdown` (`cn.lineai.ui.markdown.*`), using `org.commonmark` + GFM tables (the only third-party Java deps alongside `org.json` and `jsch`). Tool-call cards live in `cn.lineai.ui.component.toolcall.*` (`ToolCallReadView`, `ToolCallWriteView`, `ToolCallShellView`, `ToolCallAgentView`, `ToolCallAgentPipelineView`, `ToolCallGenericView`, etc.) — wire any new tool through `ToolCallUtils` so its summary card renders correctly.

### Theming
`LineTheme` (`:ui-theme`) holds the active `ThemePalette` as mutable static fields (updated by `LineTheme.apply(palette)`). Nine built-in palettes plus one custom palette are available; the custom palette exposes 19 editable colour keys via `ThemePalette.EDITABLE_KEYS` (including `KEY_SURFACE`, `KEY_TEXT_ON_COLOR`, `KEY_BORDER_LIGHT`, `KEY_CODE_BORDER`). `ThemeSettingsScreenView.COLOR_FIELDS` mirrors the same order. `IconButtonView.drawableFor(int)` uses a `SparseArray<Integer>` registry (no `switch`); add new icon constants by appending to the static initialiser block.

## Conventions

- Java only in `app/src/main/java` — do not add Kotlin sources. The Kotlin stdlib is explicitly excluded from the runtime classpath in `app/build.gradle.kts`; pulling in a Kotlin transitive will silently fail at runtime.
- Views are constructed in Java code, not inflated from XML layouts. `lint.xml` ignores `ViewConstructor` and `IconDuplicates` deliberately.
- Tests live under `app/src/test/java/cn/lineai/...` mirroring the production package; JUnit 4 + `org.json` only (no Robolectric, no Mockito). When testing repository or controller logic, prefer in-memory fakes that already exist in sibling tests. Feature modules have their own tests under `feature-*/src/test/java`.
- Never hardcode user-visible Chinese or English strings in Java — add them to `app/src/main/res/values/strings.xml` and `app/src/main/res/values-zh/strings.xml` and load via `R.string.*` (or `cn.lineai.ui.theme.R`/`cn.lineai.ui.markdown.R` for resources owned by those modules). Watch for `\uXXXX`-escaped Chinese that bypasses greps.
- `问题.txt` is a scratch issue list and is gitignored; do not edit it as part of normal changes.

## Architecture Principles

### Open-Closed Principle (OCP)
- 新增工具只需覆写 BaseTool 的 getDisplayCategory()/getActionName()/getActionIcon() 虚方法，无需修改 ToolDisplayResolver/ToolCallReadView/ToolCallBlockView 等已有代码
- 新增协议通过 ModelProtocolFactory.register() 和 ModelCatalogClient.register(ModelProtocolType, CatalogFetcher) 扩展，无需修改工厂或客户端
- 协议能力判断（supportsDedicatedCompression 等）由 ModelProtocolType 枚举属性承载，不在 ModelConfig 中硬编码
- ArchiveSecretRedactor 敏感字段和命名模式通过 registerSensitiveFields()/registerSensitiveNameKeyword() 实例方法扩展
- UrlPolicy 白名单主机和网段通过 addCleartextHost()/addPrivateNetworkPredicate() 实例方法扩展
- Agent 类型→模板映射通过 AgentPromptBuilder.registerRoleTemplate() 静态方法扩展
- Agent 工具白名单通过 ToolSettingsStore.getAgentExcludedToolNames()/getAgentAllowedCategories() 配置

### Single Responsibility Principle (SRP)
- Repository 仅负责数据存取：ToolSettingsRepository 仅存取配置，权限判断在 ToolPermissionService，提示词渲染在 ToolPromptService
- LearningContextRepository 仅存取数据，业务编排（buildLearningContext/getOverview）在 LearningContextService
- DiffRepository 仅管理 DB 记录，文件恢复在 FileRestorer，DB 标记在 DiffRepository.markReverted()
- ChatModeRepository.applyMode() 仅存储模式，权限同步由调用方通过 applyPermissionForMode() 处理
- ToolExecutor 仅做执行编排，Diff 记录在 DiffRecorder
- Protocol 实现仅做组合编排：OpenAiMessageSerializer 负责消息序列化，CodexRequestBuilder 负责请求构建，CodexOutputMerger 负责输出合并

### Dependency Inversion Principle (DIP)
- Repository 不依赖 UI 层 (cn.lineai.ui.*)、AI 层 (cn.lineai.ai.prompt.*)、Service 层 (cn.lineai.service.*) 的具体实现
- 解耦接口定义在 :core-api：ToolCategoryResolver、ToolPromptRenderer、AccessibilityStateProvider、SkillPromptProvider、PromptTemplateProvider
- Repository 不持有 Android Context：LineCodeDatabase/SettingsRepository/SharedPreferences 通过构造函数注入
- 资源访问通过 ResourceProvider 接口（openAsset/getString），系统配置通过 SystemConfigProvider 接口（isDarkModeEnabled/getSdkInt/getFilesDirPath）
- feature-model 对 data 层的依赖通过 PromptTemplateProvider 接口解耦

### OOP Principles
- View 层仅负责渲染：不包含数据解析、业务验证、ModelConfig 构建、后台线程启动、网络请求等逻辑
- BaseTool 通过虚方法提供显示信息（getDisplayCategory/getActionName/getActionIcon），UI 层不硬编码 if-else 分派
- PhoneControlService 接口解耦 Phone 工具与 LineCodeAccessibilityService

### KISS Principle
- 不创建仅被单一调用方使用的抽象层
- 不引入项目中没有其他消费者的设计模式
- Protocol 中 SSE 解析逻辑保留在 Protocol 内（与流处理紧耦合，强行提取增加间接层但不增加可测试性）
- ToolExecutor 中权限检查保留（仅2行条件判断，提取反而增加间接层）

### Do One Thing
- 每个方法只做一件事：ChatModeRepository.applyMode() 不执行跨 Repository 副作用
- DiffRepository.revertDiff() 不写文件，文件恢复由调用方通过 FileRestorer 处理
- render() 方法不执行状态变更副作用

### Internationalization (i18n)
- 所有用户可见字符串必须使用 R.string.* 资源，禁止在 Java 代码中硬编码中文或英文字符串
- 英文资源在 values/strings.xml，中文资源在 values-zh/strings.xml，俄文资源在 values-ru/strings.xml
- 警惕 \uXXXX 转义的中文字符，这些绕过 grep 搜索但不违反规则
- AI 提示词中的中文关键词（如 MemoryExtractionService 的记忆匹配词、SkillFileManager 的 Skill 模板）不属于用户可见字符串，不需要 i18n
- Repository 层通过 ResourceProvider.getString() 获取字符串资源，不直接使用 Context
- Agent 提示词 fallback 通过 Context.getString() 读取资源，无 Context 时回退到硬编码英文

## Compliance gates before shipping a release

`docs/android-compliance-audit.md` is the source of truth. Before tagging, the following must pass: `:app:testDebugUnitTest`, `:app:lintDebug`, `:app:assembleDebug`, `:app:assembleRelease`. Debug signing must never sign release artifacts — the `validateReleaseSigning` task enforces this.
