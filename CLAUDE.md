# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

LineCode Pro (`cn.lineai`, applicationId `cn.lineai`) is an Android app that hosts an in-device AI coding workspace: an LLM chat with a tool-call loop over user-selected project trees, plus optional SSH/Termux integration. Single-module Gradle project (`:app`), pure Java sources (no Kotlin in `app/src`), Android Gradle Plugin 9.2.0, `compileSdk = 36`, `minSdk = 24`, `targetSdk = 36`, Java 11.

## Commands

All Gradle commands go through the wrapper. Settings force `FAIL_ON_PROJECT_REPOS`, and the repo is wired to a Tencent Cloud Maven mirror — keep that in `settings.gradle.kts` when adding repositories.

- Debug build: `./gradlew :app:assembleDebug`
- Debug build signed with debug cert for sideload (separate APK output): `./gradlew :app:assembleDebugUserCert` (also `exportDebugUserCertApk` copies it to `app/build/outputs/apk/debugUserCert/export/LineCode-user-cert-debug.apk`)
- Release build: `./gradlew :app:assembleRelease` — fails fast via `validateReleaseSigning` unless `signing.properties` (gitignored) contains `storeFile`, `storePassword`, `keyAlias`, `keyPassword`. On success, `exportReleaseApk` renames the artifact to `LineCode Pro <versionName>.APK` under `app/release/`, then `purgeReleaseSymbolFiles` deletes `outputs/mapping/release` and `outputs/native-debug-symbols/release`. The release uses an 8192-entry obfuscation dictionary generated by `generateReleaseObfuscationDictionary` into `app/build/generated/r8/obfuscation-dictionary.txt`.
- Unit tests: `./gradlew :app:testDebugUnitTest`
- Single test: `./gradlew :app:testDebugUnitTest --tests "cn.lineai.tool.ToolRegistryTest"` (append `.methodName` for one method)
- Lint: `./gradlew :app:lintDebug` (rules in `app/lint.xml`)

Bumping the version requires editing `releaseVersionName` and `defaultConfig.versionCode` in `app/build.gradle.kts`; the release APK filename is derived from `releaseVersionName`.

## Architecture

### MVP coordinator
`MainActivity` is the only Activity. It instantiates `MainCoordinator` (the presenter), constructs `MainChatView` programmatically (UI is built in Java code, not inflated from XML — `lint.xml` silences `ViewConstructor` accordingly), and forwards SAF picker / permission results. The coordinator implements `MainUiController` and delegates large chunks of behavior to per-concern controllers in `cn.lineai.mvp.*` (e.g. `ChatController`, `GenerationController`, `ToolRunController`, `ModelManagementController`, `SettingsController`, `ScreenNavigationController`, `PermissionModeController`, `LineCodeArchiveController`, `ProjectSheetController`, `SshFileTreeController`, `FileOperationController`). `MainDependencies` wires repositories, services, and the tool stack into the coordinator. UI state flows through `ChatUiStateAssembler` → `ChatUiState` → `MainContract.View.render(...)`. When adding new chat/tool features, prefer extending an existing controller and threading state through the assembler rather than reaching into views directly.

### Model protocol abstraction
`cn.lineai.ai.protocol.ModelProtocol` is the streaming/completion interface. `ModelProtocolFactory` dispatches on `ModelProtocolType`:
- `OPENAI_COMPATIBLE` → `OpenAiCompatibleProtocol` (default; also see `OpenAiCompatibleCapabilities`, `OpenAiResponsesCompactionProtocol`)
- `CODEX_RESPONSES` → `CodexResponsesProtocol` (uses `ResponsesInputBuilder`)
- `ANTHROPIC_MESSAGES` → `AnthropicMessagesProtocol`
- `LOCAL_GGUF` → `LocalGgufProtocol`

Messages flow via `cn.lineai.ai.message.*` (`SystemModelMessage`, `UserModelMessage`, `AssistantModelMessage`, `ToolModelMessage`). `ModelClient` is the high-level entry point used by `MainCoordinator`. Tool-call text inside a stream is parsed by `ToolCallTextParser`; reasoning by `ThinkTagParser`. System prompts are assembled by `cn.lineai.ai.prompt.SystemPromptProvider` from `app/src/main/assets/prompts/*.txt` templates (`system-prompt-template.txt`, tone variants, `context-compaction-template.txt`, `memory-extraction-template.txt`, `skill-extraction-template.txt`, `work-directory-template.txt`, `learning-context-template.txt`, `model-identity-template.txt`) — modify templates there, not by hardcoding in Java. `SystemPromptProvider.build(..., ModelConfig model)` renders the model identity block (modelId / name / provider / protocol) from the currently selected model and injects it as `{{MODEL_IDENTITY}}` so the assistant can answer capability questions against the actual model id.

### Tool system
`BaseTool` (name, description, category, JSON schema, `execute(JSONObject, ToolContext)`) is the contract. `ToolRegistry` registers built-ins in `cn.lineai.tool.builtin.*` at construction (`FileReadTool`, `FileWriteTool`, `FileEditTool`, `FileDeleteTool`, `GlobTool`, `ListDirectoryTool`, `AgentTool`, `AgentPipelineTool`, `ShellExecuteTool`, `ImageUnderstandingTool`, `ImageGenerationTool`, `WebSearchTool`, `WebFetchTool`) and then loads user extensions with `agentx_` / `mcpx_` name prefixes via `reloadExtensions()` (sourced from `ExtensionRepository` → `CustomAgentExtensionTool`, `CustomMcpHttpTool`). Tool execution is driven by `ToolExecutionCoordinator` + `ToolExecutor`; user confirmation gating goes through `PermissionModeController` and `ToolReviewListener`, with session-scoped auto-confirmation tracked in `MainCoordinator.sessionAutoConfirmedTools`. File-touching tools route paths through `FileToolPathPolicy`.

### Context window management
`ContextManager` estimates token usage with the constants `CHARS_PER_TOKEN = 4` and `MESSAGE_OVERHEAD_TOKENS = 8`, reserving `DEFAULT_RESERVE_TOKENS = 2048`. When the snapshot signals overflow, `ContextCompactionService` summarises older messages using the active model and `context-compaction-template.txt`. `MemoryExtractionService` and `LearningContextRepository` feed durable knowledge back into the system prompt.

### Storage
SQLite is centralised in `cn.lineai.data.db.LineCodeDatabase` (singleton, version 1) with schema in `LineCodeSchema` (tables for projects, models, conversations, messages, blocks, tool calls/results, attachments, diffs, memories, working memory, skills/usage, extension agents/MCPs, import jobs). Repositories in `cn.lineai.data.repository.*` are the only callers — do not open the DB directly from controllers. Settings repositories (`SettingsRepository`, `InputSettingsRepository`, `OutputSettingsRepository`, `ThemeSettingsRepository`, `AiBehaviorSettingsRepository`, `ToolSettingsRepository`, etc.) follow the same pattern.

### Project workspace & files
Workspace roots come from user-picked SAF trees. `WorkspacePaths` resolves on-device paths, `SafPathResolver` translates between SAF document URIs and POSIX-like paths, and `StoragePermissionManager` drives the `MANAGE_EXTERNAL_STORAGE` / legacy storage prompts (the manifest requests both because the app operates as a coding workspace over external project trees). SSH access goes through `SshService` (jsch) and `SshFileTreeRepository`.

### Security / URL policy
All outbound URLs (in-app browser, external link opening, model HTTP, model catalog, web fetch/search, custom MCP HTTP) must pass through `cn.lineai.security.UrlPolicy`. HTTP-only protocols, and cleartext is restricted to `localhost`, `127.0.0.1`, `10.0.2.2` (mirrored in `res/xml/network_security_config.xml`, `cleartextTrafficPermitted="false"` by default). The built-in WebView disables file/content access, blocks mixed content, and keeps JavaScript off unless the user explicitly toggles it. When exporting `.linecode` archives, `ArchiveSecretRedactor` strips model API keys, SSH passwords / private keys / passphrases, web search API keys, and sensitive MCP headers — extend it whenever you add a new secret-bearing field. See `docs/android-compliance-audit.md` for the documented threat model and accepted risks.

### Markdown & custom views
The chat list renders Markdown through `cn.lineai.ui.markdown.*`, using `org.commonmark` + GFM tables (the only third-party Java deps alongside `org.json` and `jsch`). Tool-call cards live in `cn.lineai.ui.component.toolcall.*` (`ToolCallReadView`, `ToolCallWriteView`, `ToolCallShellView`, `ToolCallAgentView`, `ToolCallAgentPipelineView`, `ToolCallGenericView`, etc.) — wire any new tool through `ToolCallUtils` so its summary card renders correctly.

## Conventions

- Java only in `app/src/main/java` — do not add Kotlin sources. The Kotlin stdlib is explicitly excluded from the runtime classpath in `app/build.gradle.kts`; pulling in a Kotlin transitive will silently fail at runtime.
- Views are constructed in Java code, not inflated from XML layouts. `lint.xml` ignores `ViewConstructor` and `IconDuplicates` deliberately.
- Tests live under `app/src/test/java/cn/lineai/...` mirroring the production package; JUnit 4 + `org.json` only (no Robolectric, no Mockito). When testing repository or controller logic, prefer in-memory fakes that already exist in sibling tests.
- `问题.txt` is a scratch issue list and is gitignored; do not edit it as part of normal changes.

## Compliance gates before shipping a release

`docs/android-compliance-audit.md` is the source of truth. Before tagging, the following must pass: `:app:testDebugUnitTest`, `:app:lintDebug`, `:app:assembleDebug`, `:app:assembleRelease`. Debug signing must never sign release artifacts — the `validateReleaseSigning` task enforces this.
